你看：Webhook 为什么看起来是"标准答案"

说白了，你查飞书官方文档、看网上教程，99% 都在说怎么配 Webhook。绝大多数 IM 平台（钉钉、企业微信）确实都是 Webhook 模式，所以开发者会默认觉得"这是行业标准"。

但问题来了：

- 你需要一个公网 IP 或域名（本地开发？不好意思，ngrok 救命）

- 你需要装 ngrok，配 authtoken（额外依赖）

- ngrok 免费版隧道不稳定，重启后 URL 就变了，每次都要跑回飞书开发者后台更新一遍

- 你还得处理 Verification Token 和 Encrypt Key，多一层加密配置

- 最绝的是：当 ngrok 隧道断开时，飞书的消息就完全收不到

你想想，你在 Webhook 模式下开发，突然网络抖动，隧道掉了，你还没反应过来，已经损失了 10 条消息。这在生产环境有多坑？

其实：长连接模式才是降维打击

飞书有个长连接（WebSocket）模式，是专门为了本地开发和内网环境设计的。听我列这些优势：

- 无需公网 IP、无需 ngrok：飞书 SDK 主动建立 WebSocket 连接到飞书服务器，你的电脑根本不暴露在网络上

- 配置极简：只需要 App ID 和 App Secret，没有 Verification Token、没有 Encrypt Key、没有 Port 折腾

- 自动重连：连接掉了，SDK 自动重连，不会像 ngrok 一样隧道没了就彻底黑了

- 开发友好：你在笔记本上写代码，飞书机器人照样实时收消息，零延迟

来看对比，我写个表格给你感受一下差别有多大。

Webhook 模式（踩坑路线）：1. 安装 ngrok；2. 设置 authtoken；3. 启动隧道；4. 复制 HTTPS 地址；5. 跑到飞书后台粘贴；6. 填 Token 和 Encrypt Key；7. 等验证；8. ngrok 崩了？重新来。

长连接模式（降维路线）：1. 飞书后台选长连接；2. 填 App ID 和 Secret；3. 点启动。完事儿。

你数一下，一个 8 步，一个 3 步。最关键的是，长连接掉线自动重连，Webhook 掉线就是黑屏。

说白了：你把简单的事情复杂化了

为什么这么多人还在用 Webhook？原因很简单：

- 第一，信息不对称。网上教程大多数讲 Webhook，没人专门拿出来讲长连接的好处

- 第二，路径依赖。你看别人怎么做的，就跟着怎么做，没人问这是最优解吗

- 第三，认知惯性。既然其他 IM 都是 Webhook，飞书也应该是——错了，飞书考虑周到，给你留了更好的方案

这就是为什么我说这是个反常识的地方。看似"标准"的 Webhook，反而是负担。真正聪明的做法，是用飞书设计好的长连接。
