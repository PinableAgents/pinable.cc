打破全局限制，模型选择自由

hi,我是PinableAI。

今天要聊一个很多开发者都在吐槽的问题：AI 工具的模型选择太不自由了。

你有没有遇到过这种情况？正在写代码用的是 Claude 模型，结果另一个项目需要用 Codex，你得去设置里改全局配置。这一改不要紧，所有会话都跟着变了。最坑的是，有时候模型失效了，你必须跑到 Provider 页面去修改 active group，直接影响所有正在进行的任务。

这就是所谓的「全局 active group」问题——一个设置绑死所有会话。

分组架构：每个会话独立选择

pinable-desktop 最新提交 d2a7b19 解决了这个痛点。核心思路很简单：把模型分组的定义和调用分离。

Provider 页面依然负责管理分组定义（也就是数据源），但在 WorkflowsPage、GatewayPage、TerminalPage 启动时，你可以独立选择 Group + Model。每个会话都有自己的环境变量，互不干扰。

后端架构也配合做了调整：

- 新增 BuildWrapperEnvForGroup() 方法，支持 per-session env 注入，优先级高于宿主进程 env

- Terminal Service 引入 Functional Options 模式，通过 WithEnvOverrides() 注入个性化环境变量

- Gateway Service 在 handoff 时自动注入对应 group 的环境变量

简单说就是：以前是一把钥匙开所有锁，现在每把锁都有自己独立的钥匙。

实际体验：哪里需要就选哪里

具体使用场景是这样：

1. Workflows 工作流：启动时三级联动——Backend → Group → Model，独立选择本次工作流使用的模型分组

2. Gateway 网关：Defaults Group 独立配置，不同项目用不同的模型

3. Terminal 终端：直接指定 providerID + groupID 启动，每个终端会话都可以用不同的模型

模型失效了？不用再跑回 Provider 页面改全局配置，直接在当前会话切换分组就行。这就是真正的「按需选择」。

写在最后

这个功能看似小改动，实际意义很大。它把「模型选择权」从系统级别下沉到了会话级别。对于需要同时处理多个项目的开发者来说，这意味着你可以：

- A 项目用 Claude 3.5

- B 项目用 Codex

- C 项目用 Gemini

完全并行，互不影响。这才是真正的效率工具。
